주제핵심

0. 이 도구가 웹 클로드와 뭐가 다른가

도구가 여러 개지만 입구는 하나 — 터미널의 클로드 한 곳에서 전부 연결됩니다.

🧩 하나의 클로드 안에 붙은 도구들
도구역할
legal-terminal사건폴더·zip 통째로 읽고 작업
legal-books저작권 청정 자료로 만든 내 서재(검증용)
korean-law · 법고을법령 조문 원문 · 판례 검색
jurisupport사건 상태·기일·증거·할일 관리
brief-protocol준비서면 단계 오케스트레이션
→ 전부 하나의 클로드 안에서 연결됩니다. 도구가 많아 보여도 쓰는 입구는 터미널 한 곳뿐.
📂 13개 파일을 안 올렸는데 다 읽는다
웹 클로드는 파일을 일일이 업로드해야 합니다. 여기선 사건폴더만 가리키면 알아서 전부 읽습니다 — 이게 결정적 차이.
웹 Claude 파일 하나씩 업로드 · 13개면 13번
legal-terminal 폴더만 가리키면 13개 통째로 + 법령·판례·과거 사건까지 연결
💬 "코웍(Cowork) 쓰면 되지 않나?"
Cowork도 로컬 파일은 읽지만 Max 전용·research preview·비개발자용 단순 UI입니다. MCP 연동(법령·판례)·skills·멀티세션 병렬·brief-protocol 오케스트레이션은 클로드코드(legal-terminal) 영역.
🪟 여러 사건도 동시에
창을 여러 개 띄우면 사건도 여러 건 동시에 굴러갑니다. 혼자 하면 반나절씩 걸릴 일을 병렬로.

1. 잘 시키는 5원칙

결과가 안 좋으면 십중팔구 명령이 부실해서입니다. 송무 지시서 쓰듯 시키세요.

① 맥락을 먼저
모델은 내 사건을 모릅니다. 맥락이 없으면 일반론·환각으로 메웁니다.
✗ "대여금 답변서 써줘" → 교과서 템플릿
✓ 사건폴더를 읽히고 "이 사건 기준으로" → 그 사건 답변서
② 역할·입장 지정
같은 사실도 입장 따라 강조점이 달라집니다.
✓ "원고(임차인) 대리인 입장에서" — 안 주면 어정쩡한 중립 글
③ 산출물 형식 명시
표·항·분량·저장파일을 못박으면 매번 같은 결과(재현성↑).
✓ "호증번호/서증명/입증취지/작성일 열로 표" → 증거설명서에 바로 붙음
④ 한 번에 다 말고 단계로
작업을 쪼개면 단계별 정확도가 오르고, 중간에 교정할 수 있습니다.
✓ 쟁점 추출 → 목차 → 초안 — "준비서면 다 써줘"보다 안전
⑤ 틀리면 되물어라
모델은 추론을 설명·수정할 수 있습니다. 단, 최종 검증 책임은 변호사.
✓ "이 인용 출처 표시하고, 불확실하면 표시해"
⚖️ 나쁜 명령 → 좋은 명령 (Before / After)
✗ "이 사건 답변서 써줘"
맥락·입장·형식이 다 빠짐 → 일반 템플릿
✓ "이 폴더(차용증·문자·입금내역) 다 읽고 · 피고 대리인 입장에서 · 변제·소멸시효 항변 중심으로 · 쟁점별 항 나눠 1~2페이지, 답변서.md로 저장"
맥락 + 입장 + 형식을 다 박음 → 그 사건 답변서
좋은 명령 = 맥락 + 입장 + 형식 · 나머지는 대화로 교정

2. 업무별 '잘 시키는 법' (도구 × 예시 명령)

우리 사무소에 깔린 도구를 업무에 1:1로 붙인 것입니다. 예시 명령은 복사해서 바로.

1 사건 파악 — legal-terminal(폴더 통째)
이 사건폴더 전부 읽고 ① 사실관계 시간순 표, ② 쟁점 목록,
③ 우리(피고)에게 불리한 점 순으로 정리해줘.
2 법령 조사 — korean-law(조문 원문)
주택임대차보호법 제6조의3 현행 조문 원문과
직전 개정 전 조문을 표로 대비해줘.
3 판례 검색 — 법고을(무료)·lbox(유료 직접)
임차인 원상회복의무 범위에 관한 대법원 판결을 법고을에서 찾아
사건번호·선고일·판시사항을 표로, 출처 URL까지.
※ 판결번호는 1차 후보일 뿐 — 인용 전 반드시 진본 확인.
4 내 판단 검증 — legal-books(교과서 대조)
'일부변제는 채무 전부의 묵시적 승인으로 시효가 중단된다'는
내 판단이 통설·판례와 맞는지 legal-books에서 대조하고,
어긋나거나 예외가 있는 부분만 짚어줘.
5 과거 사건 참고 — case-records
유사한 대여금 사건에서 우리가 변제·시효 항변을 어떻게 구성했는지,
법원이 어떻게 판단했는지 과거 기록에서 찾아줘.
6 준비서면 — brief-protocol(단계)
brief-protocol로 진행 — 쟁점 추출 → 법령(korean-law)·판례(법고을)
→ 목차 승인받고 → 초안. 각 단계 끝나면 멈추고 확인받아.
7 제출 전 자기검증 — mock-hearing
이 준비서면 초안을 상대방 대리인·재판부 관점에서 공격해줘.
가장 약한 전제부터 순서대로, 보강 과제까지.
8 사건관리·기일 — JuriSupport / hearing-check
2주 이내 기일이 있는 사건과 미완료 할일을 마감 임박순으로 표로 정리해줘.
※ 파일은 OneDrive, 상태·기일·할일은 JuriSupport로 역할분담. JuriSupport 없이 CSV 한 파일로도 관리 가능(songmu-legal case-index).

3. 내 법률 서재(legal-books) — 무엇을 넣어도 되나

  1. 판례·결정문 전문 (저작권법 §7 3호 보호 제외) — 가장 안전
  2. 법령·행정규칙 전문 (§7 1·2호)
  3. 공공기관 무료 해설·실무자료 (공공저작물 자유이용 §24-2)
  4. 변호사 본인 저작물(서면·메모·블로그) — 가장 깨끗 + 사무소 노하우 누적
  5. CC 라이선스·저작권 만료 고전
⚠️ 상용 교과서·단행본 — 본인 업무용 참고·검색은 가능하지만, DB(복제물)를 남에게 배포·공유하는 것은 위험. (개인 이용 ≠ 배포)

4. 검증은 변호사 (가장 중요)

AI의 가장 큰 위험은 그럴듯한 가짜 판결번호(환각)입니다. 제출 전 반드시:

  1. 인용 판결번호 실존 확인 — 법고을 / lbox에서 직접 검색
  2. 직접인용("…")은 원문과 글자 단위 대조
  3. 법령 조문번호는 korean-law / law.go.kr로 재확인
⚡ 실제 사례 — 2025. 7. 24. 대법원 전원합의체(2023다240299)가 "시효완성 후 채무승인 = 시효이익 포기 추정"이라는 수십 년 된 법리를 폐기했습니다(이제 개별 의사해석). AI 학습데이터엔 옛 법리가 더 깔려 있을 수 있어, 그대로 받아쓰면 폐기된 법리로 서면을 씁니다. → 현행 판결을 직접 확인하고 검증하세요.
